A medium-length block post

This fixture is roughly three kilobytes of body content, designed to stress parse_blocks and the render-time kses pass with enough block variety to matter without becoming an outlier. It contains headings, paragraphs, lists, a quote, an embedded code block, a table, and a group with columns — the building blocks (pun intended) of a typical long-form blog post before it turns into a wall of text.

Why realistic shapes matter

WordPress ships a block grammar parser that's quadratic in some pathological cases but linear for well-formed content. Patina's Rust parser is the same complexity class but with a much smaller constant factor. The test here is not "does it beat the PHP parser on a degenerate input" — it's "does it beat the PHP parser on content that looks like what real users write". Hence: multiple block types, a mix of nesting depths, and some inline formatting.

  • Shapes matter more than sizes
  • Block variety beats repetition
  • Real users mix markup and prose freely
  • Comment-heavy posts render kses on every comment body
  • Search results hit the same scenario tree

"If your benchmark doesn't look like production, it's not a benchmark — it's a microbenchmark, and microbenchmarks lie by amortizing away the exact costs you came to measure."

— Every performance engineer ever

Column one holds a short note about what a two-column group looks like in the block tree — one core/group, one core/columns, two core/column children, and whatever lives inside them.

Column two is the counterweight. Nesting depth goes from zero (the root) to four (group → columns → column → paragraph), which is plenty to exercise the recursive descent in parse_blocks without getting silly.

Finally some inline-heavy text with bold, italics, inline code, a link, an email, and entities like &mdash; — all of which flow through the render-time kses pass.
