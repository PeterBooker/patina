A medium block post — thirty flat blocks of realistic content

WordPress block content usually arrives at parse_blocks as a flat sequence of top-level blocks separated by HTML comment delimiters. The parser walks the string once, emits a node per delimiter pair, and recurses only when it sees an opening marker without a matching close on the same nesting level. Flat content is the overwhelmingly common shape in practice — most posts are paragraph-heavy with the occasional heading, list, and quote, and very little structural nesting outside of templates and patterns.

The shim layer Patina uses to intercept core functions adds roughly five hundred nanoseconds of overhead per call. That figure is noise next to the multi-millisecond body of wp_kses or parse_blocks, but it becomes observable if the override is called tens of thousands of times per request. The mitigation, when it matters, is to push the hot loop down into Rust so the shim boundary is crossed once per block tree rather than once per block.

Escaping is surprisingly expensive in aggregate. esc_html and esc_attr are each called hundreds of times during a typical page render — every template tag, every attribute on every link, every translated string that might contain user data. Individually they're cheap, but cumulatively the PHP version spends a measurable slice of wall-clock time in htmlspecialchars and its zend_string allocation bookkeeping.

Section 1: more ground to cover

Comment-heavy pages are where wp_kses earns its cost. Every approved comment body is run through the allowed-tag filter at render time, and the allowed-tag set for comments is large enough that the PHP implementation walks a non-trivial state machine across each comment. A post with fifty comments renders kses fifty times, and the cost scales linearly with both comment count and average comment length.

  • Paragraph blocks dominate the distribution and absorb most of the parser's work
  • Heading blocks are cheap individually but accumulate over document structure
  • List blocks expand into non-trivial HTML once the allowed-tag filter runs on them
  • Quote blocks are uncommon per post but carry heavy inline markup when they appear
  • Code blocks skip most of the kses pipeline because 
     is largely pass-through
  • Table blocks are rare but exercise the deepest branch in the block grammar

Tables of contents, glossaries, and reference sheets push posts into the long tail of body size. These posts are rare — under five percent of a typical corpus — but they dominate the p95 latency on pages that render full content. The parser spends proportionally more time on them, which is exactly why they're included in the benchmark corpus: the long tail is where optimizations pay back.

"If your benchmark doesn't look like production, it's not a benchmark — it's a microbenchmark, and microbenchmarks lie by amortizing away the exact costs you came to measure."

— Every performance engineer, eventually

Real-world block usage is heavily zipfian. Paragraphs account for roughly seventy percent of all blocks in published WordPress content, headings and lists together for another twenty, and everything else — quotes, code, images, tables, embeds, buttons, spacers, separators — shares the remaining ten. A benchmark corpus that over-represents the tail will measure a parser's dispatch fairness; one that tracks the zipf will measure the thing users actually hit.

fn parse_blocks(input: &str) -> Vec<Block> {
    // Recursive descent over the block grammar.
    // Byte-for-byte match with class-wp-block-parser.php.
    let mut parser = Parser::new(input);
    let mut out = Vec::new();
    while let Some(block) = parser.next_block() {
        out.push(block);
    }
    out
}

The grammar WordPress uses for blocks is recursive descent over HTML comments with a JSON attributes blob on the opening marker. It parses in linear time on well-formed input and degrades to quadratic only on pathological cases involving unclosed blocks — which the editor prevents from ever landing in the database, so in practice the linear branch is the only one worth optimizing.

Inline markup is another hidden cost. A paragraph with a dozen inline tags — bold, italic, code, links, spans — generates that many additional nodes in the DOM the kses walker has to validate. The test fixtures below deliberately sprinkle inline markup into each paragraph so the render-time sanitizer does non-trivial work on every block, rather than short-circuiting on a plain string of prose.

Entities and character references exercise a different code path in the escaper than raw ASCII does. Ampersands, less-than signs, quotes inside attribute values, and the occasional Unicode emoji all exercise branches that plain text skips. The corpus sprinkles them throughout so the escape benchmarks don't accidentally measure the fast path exclusively and overstate the real-world win.

Section 2: more ground to cover

Full-site-editing templates wrap post content in a stack of group, columns, and query blocks before the inner HTML ever reaches the render loop. The template overhead is constant per request — it doesn't scale with post size — so its cost amortizes across anything that saves time inside the post body. That amortization is why the long fixture below matters most: the savings on one big post dwarf the template-wrapper overhead.

  • Code blocks skip most of the kses pipeline because 
     is largely pass-through
  • Table blocks are rare but exercise the deepest branch in the block grammar
  • Separator blocks are pure structure and parse in constant time
  • Group blocks are the primary nesting container in modern themes
  • Columns blocks always contain two or more column children and double the descent depth
  • Image blocks carry alt text through kses on every render

Cache behavior dominates microbenchmark variance. The same request served from a warm opcache with a warm MySQL query cache and a warm filesystem cache can be twice as fast as the same request on a cold start. The bench runner deliberately warms all three before starting the measurement window, so the numbers reflect steady-state rendering rather than cold-start noise.

"Depth matters more than count. A flat list of a hundred paragraphs is easier to parse than ten paragraphs wrapped in five layers of groups."

— Folk wisdom from the block editor team

The practical ceiling on parse_blocks performance is the time WordPress spends in the database and template layer. Even if the parser took zero nanoseconds, the request floor is bounded by MySQL round trips, PHP opcode execution, and template rendering. The Rust wins show up as tail-latency improvements on content-heavy pages, not as median improvements on the homepage.

Good benchmarks look like production. That means realistic post shapes, realistic comment densities, realistic template complexity, and realistic query patterns. A corpus of one thousand identical posts generated from a loop will measure a cache, not a renderer. The fixtures in this directory aim for variety within each tier — different block types, different inline density, different content shapes — so the parser exercises all its branches.

Regressions hide in the places you don't measure. Adding a new override without a filter-compatibility integration test is how a seemingly-equivalent Rust implementation ends up silently diverging from WordPress output on some obscure wp-admin screen. Every shimmed override in this project ships with an integration test that calls it through real WordPress hooks, not a unit test that bypasses them.

The measurement goal is tail latency, not median. Median request time on a warm WordPress install is dominated by fixed costs — MySQL, template boot, plugin init — that Patina does not touch. The interesting deltas are at the p95 and p99 of content-heavy scenarios, where the Rust overrides save enough wall-clock time to move the tail while leaving the median unmoved.
