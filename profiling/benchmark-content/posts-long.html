A long-form block post — the parse_blocks win zone

This fixture targets eight kilobytes of body content with deep block nesting — the scenario where parse_blocks spends real time and therefore where a native Rust replacement should show the biggest absolute win. Eight kilobytes is the upper tail of "normal" blog content: long opinion pieces, tutorial walkthroughs, detailed release notes, conference write-ups. Content above this size is rare enough in practice that optimizing it further has diminishing real-world impact.

Section one — nesting for days

This whole section is wrapped in a core/group. Groups are the most common nesting block in modern WP themes — they're how FSE patterns compose, how cover blocks contain their children, and how full-site templates wrap entry content. If the Rust parser is slow on groups, it's slow on everything.

Left rail

Column one is the left rail. In a realistic theme this would hold a table of contents, or maybe a sticky author card. We use it to hold more text because the goal is byte-count pressure, not design fidelity.

  • List item one — deeply nested inside column one inside a columns block inside a group
  • List item two with a nested link
  • List item three with inline code and bold

Right rail

Column two is the right rail. Four levels deep: group → columns → column → paragraph. This is the nesting depth the real-world corpus hits most often — deeper than this and you're in patterns/templates, not post content.

"Depth matters more than count. A flat list of 100 paragraphs is easier to parse than 10 paragraphs wrapped in 5 layers of groups."

Section two — block variety

The block grammar has roughly a dozen core blocks that appear in nearly every post. Below we cycle through them so the Rust parser exercises every branch of the top-level dispatch: paragraph, heading, list, quote, code, preformatted, table, separator, spacer, image caption, group, columns. The pattern repeats because block frequency is heavily zipfian — paragraphs dominate by a factor of ten, lists and headings are the next tier, and everything else is a long tail.

Another paragraph. Patina's shim layer adds one PHP frame per call, which shows up as ~500 nanoseconds of overhead per parse_blocks invocation — effectively free next to the multi-millisecond body of the function itself. The worry isn't shim overhead on this path; the worry is whether the Rust parser matches WP's exact output shape including innerContent placeholders.

  • Paragraphs are the most common block
  • Lists are the second most common
  • Headings are the third
  • Everything else is a long tail of occasional use
  • Groups and columns dominate the nesting distribution
  • Cover blocks appear once per post when used at all
  • Embeds round out the top ten

"If you parse the ninety-fifth percentile post in under a millisecond, you've already won — the remaining five percent are rounding error on the average pageload."

fn parse_blocks(input: &str) -> Vec<Block> {
    // Recursive descent over the block grammar.
    // Byte-for-byte match with wp-includes/class-wp-block-parser.php.
    let mut parser = Parser::new(input);
    let mut out = Vec::new();
    while let Some(block) = parser.next_block() {
        out.push(block);
    }
    out
}

Section three — tables and closing

Closing paragraph: the long fixture is deliberately closer to the upper tail of body sizes than a typical post, because it's the scenario where parse_blocks spends enough absolute time for a speedup to dominate the per-request overhead of the extension's activation path. Measuring the win zone matters more than measuring the median.
